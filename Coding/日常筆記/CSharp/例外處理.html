    程式中之例外處理是門大學問，有時他可能微不足道，有時他也可以毀了一個應用

    剛好這幾天去上了Teddy的例外處理，這邊就記錄一下當時上課的想法

    Design By Contract

    一種開發思維，不精確來說，可以當作是在做初始值的檢查，並且把他變成是一種機制

    畢竟許多 exception 的發生，其實都跟不正確的初始值有關

    我在知乎看到了一個python範例我覺得滿淺顯易懂的[1]

    def calc_BMI(mass: float, height: float) -> float:

"""
Calculate BMI indicator value.
"""

# Contracts
assert 20 < mass    , 'Mass should not be smaller than 20[kg]. '
assert mass < 300   , 'Mass should not be larger than 300[kg]. '
assert 0.7 < height , 'Height should not be smaller than 0.7[m]. '
assert height < 2.8 , 'Height should not be larger than 2.8[m]. '

# Body
res = mass / height**2
return res

    Root Cause(根本原因)

    有的時候一個exception的發生式來源於前面code的邏輯出錯

    比方說到DB拿值，結果拿到null，但預期應該要是空陣列，這時執行foreach的code就會壞的莫名其妙，不好找到出錯源頭

    Robustness(強建度)

    可以簡單類比成系統的例外處理做的好不好，做的好，系統不容易掛掉，當然就是比較穩健的系統

    但要開發這樣的系統，成本就會非常高昂

    很多時候，開需求的人都總會覺得系統這樣那樣不應該壞掉，但給的開發時程又不是很充足

    如何跟開需求的人闡明 robustness 與 開發成本 真的是個大問題

    Fault x Error x Failure

    有的人將例外分成了3種等級，簡單來說，有點連鎖效應的感覺

    前者可會導致後者的發生，像是fault發生可能導致error也發生，error發生可能導致failure發生

    而 failure 是最危險的，因為他可能會導致服務偏離規格

    所以若要在強健度與開發時程取個平衡點的話，或許可以盡量把exception的發生控制在低一點的層級

    有時間再盡可能的去修掉可能發生exception的code

    發生來源

    其實這也是一個看待exception的角度，exception的發生要不是自己造成的，不然就是別人造成的

    若是自己造成的，控制權在自己手上，那當然就是自己能有多點的處理空間

    若是別人造成的，控制權在別人手上，有時就要祈禱對方願意改了，但若別人有歷史包袱，或兼容問題

    那狀況就比較麻煩了，可能就得由自己這邊多做些處理了

    Fault-Tolerant(高容錯)

    簡單來說，就是系統一部分壞掉了，整體還是可以正常運行

    像是微服務化的系統，可以利用像是k8s之類的技術，動態調整服務

    Asynchronous Exception

    這類型的exception還滿有趣的，像是runtime的時候遇到記憶體不足

    這類由系統主動通知的，通常都是Asynchronous Exception

    也就是說，這類Exception能不遇到就不要遇到，不然一發生似乎都是滿大條的 lol

    Try-Catch-Finally

    大多數程式語言都會有try-catch-finally這樣的語法可以用

    一般來說，大概會是這樣使用
    try 主要要執行的邏輯
    catch 有例外時要做的事
    finally 最終要做的事

    而具體來說，我大概是會這樣使用
    try 主要要執行的邏輯
    log訊息
    catch 嘗試錯誤修復
    將對外訊息整理得更具可讀性
    log訊息
    finally 最終要做的事
    關掉connection
    記憶體釋放
    log訊息

    而在上課時，我覺得投影片上的定義寫的滿不錯的
    try implement requirements(can have alternatives)
    prepare state recovery (ex make a check point)
    report exceptional conditions
    catch perform error and fault handling
    report exceptional conditions
    control retry flow
    finally release resources
    drop check points if any

    關於log

    最好不要每個try catch都寫log，否則一樣的訊息可能會被重複log好多次

    一種解法是做個"保護傘"，在最外層進行log，這樣就能確保只會log一次

    Error Handling

    error其實就已經有點可怕了，畢竟可能會造成Failure

    而針對這種情形，復原(Backward recovery)或許是種好的應對策略

    具體做法大致可分為兩類

    • State-based: 像是製作映像檔

    • Operation-based: 像是使用Undo，可以一個指令一個指令回復

    行為回復的實作策略

    Fault Handling

    • 診斷 diagnosis
    • 隔離 isolation
    • 重新組態 reconfiguration
    • 重新初始化 reinitialization

    Retry

    • 設計多樣性( design diversity )
    • 功能多樣性( functional diversity )
      • 像是不同的登入方式
      • FB登入、Google登入
    • 資料多樣性( data diversity )
      • 像是exception後給default data
    • 時序多樣性( temporal diversity )
      • 隔一段時間再retry，像是網路瞬斷

    Exception 很多是不可預期的，無法靠事先if else去做檢查

    當然能事先預判也是不錯的，畢竟Exception的產生，其實也是會耗掉系統效能的

    其實我覺得這也引導出了一個問題: 什麼是好的軟體？

    這其實在課堂上大家也有不少的討論，而有幾個方像我覺得滿不錯的

    好不好加功能?

    畢竟，不好加功能的話，那就跟硬體好像沒什麼區別了

    調整跟scope有關跟時間比較無關

    基本上來說，就是希望開發的時程是可預期的，加幾個功能大概需要花多少時間

    如果是架構很雜亂的系統，這件事一定很難達到

    甚至加一個功能搞不好還會影響到以前的功能
